Academic Best Practice

How the team applies chatbot research

The QA process is grounded in three academic frameworks from the Customer Engagement & AI module. Each specialist agent owns the dimensions relevant to their domain:

Seven-Dimension Chatbot Taxonomy

Adamopoulou & Moussiades (2020). @Fionn owns domain closure and service scope. @Dot owns goals, response generation, and channel. @Niamh owns human-aid reachability and permissions. @Siobhan ensures the UI makes each dimension visible.

CASA — Computers Are Social Actors

Reeves & Nass (1996). @Siobhan owns the disclosure-in-turn-1 bright line (badge vs spoken self-ID). @Dot owns frustration de-escalation as conversation engineering. @Niamh owns chummy-data-extraction consent (localStorage disclosure before collection). @Fionn verifies personality register matches actual capability.

Conversation Flow Anatomy

Opener → Intent → Slots → Recovery → Closer. @Dot implements the recovery ladder in code (reword → options → human). @Siobhan designs the closer UI (read-back + keep exploring + human path). @Cormac uses "design the unexpected first" as a sprint-planning rule.

@Keelin (QA) cross-references all three frameworks in every review pass, flagging findings as [DESIGN] or [CODE] so ownership is clear. @Cormac (Manager) uses framework-alignment scores as delivery quality gates.

Testing Methodology

How Bob is tested

All testing follows a real-user simulation model: tests are written as if a frustrated customer at 11pm on a Saturday is trying to use the bot. No test assumes a happy path. Every test includes a verification step that can be automated using a headless browser or simulated DOM interaction.

Test categories

The 61-case test plan (test-plan.md) covers 11 categories:

  • Greeting flow — first visit, return visit, name capture
  • Artist search — single/multi artist, misspelled, gibberish
  • Genre exploration — chips, tags, keyword search
  • Discovery panel — hero, similar artists, albums, genres, bio expander
  • Human handoff — contact details, address verification
  • Memory/persistence — localStorage, clear, CLS command
  • Voice STT — mic, transcription, auto-send
  • Voice TTS — speaker icon, playback, cancel
  • Mobile — breakpoints, layout, touch targets
  • Error handling — API failure, gibberish, fallbacks
  • UI elements — modals, buttons, badges, addresses

Automated verification

Each test case specifies what DOM element to check, what text to assert, and what network call to verify. Tests are designed to be run by a headless browser script (Puppeteer/Playwright) that:

  • Sets the viewport to the target device width
  • Simulates clicks, typing, and voice input
  • Asserts DOM state after each interaction
  • Captures screenshots on failure
  • Logs console errors as test failures
